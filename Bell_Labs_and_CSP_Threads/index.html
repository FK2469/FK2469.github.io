介绍

此页是并发编程历史的一部分, 侧重于霍尔的通信顺序进程 (CSP) [1] [1a]的特定沿袭。在这种风格的并发编程是有趣的原因不是效率, 但清晰。也就是说, 只考虑并发编程作为提高性能的一种手段,例如, 以重叠磁盘 i/o 请求, 通过预取结果到预期查询来减少延迟, 或利用多个处理器, 这是一个普遍的错误。这些优势固然重要, 但与本讨论无关。毕竟, 它们可以在其他样式 (如异步事件驱动编程) 中实现。相反, 我们对并发编程感兴趣, 因为它提供了一种自然的抽象, 可以使某些程序更简单。

这不是

大多数计算机科学专业的本科生被迫阅读安德鲁. 伯雷尔的”使用线程编程入门”SRC 线程模型是当前可用的大多数线程包所使用的。所有这些问题都是因为它们太低级了。与霍尔提供的通信原语不同, SRC 样式的线程模块中的基元必须与其他技术 (通常是共享内存) 相结合, 以便有效地使用。一般情况下, 程序员往往不构建自己的高级结构, 并且由于需要注意这样低级别的细节而感到沮丧。

现在, 把伯雷尔的教程从你的头脑中推开。这是一个不同的线程模型。如果您将其作为不同的线程模型来处理, 您可能会发现它更容易理解。

通信顺序进程

到 1978年, 在编程多的上下文中, 有许多用于通信和同步的建议方法。共享内存是最常见的通信机制, 而信号量、关键区域和监视器是同步机制之一。c.a.r. 霍尔解决了这两个问题与单一的语言原语: 同步通信。在霍尔的 CSP 语言中, 进程通过从命名的缓冲通道发送或接收值来进行通信。由于信道是缓冲的, 因此发送操作会一直阻塞, 直到该值已转移到接收器, 从而提供了同步的机制。

霍尔的示例之一是重新格式化80列卡, 以便在125列的打印机上进行打印。在他的解决方案中, 一个进程一次读取卡, 将分解后的内容字符按字符发送到第二个进程。第二个过程将组合125个字符, 将组发送到行打印机。这听起来很琐碎, 但是在没有缓冲的 i/o 库的情况下, 单进程解决方案所涉及的必要簿记是繁重的。事实上, 缓冲的 i/o 库实际上只是封装了这两种导出单字符通信接口的进程。

这些例子的线性管道性质 misrepresentative 了 CSP 的一般性质, 但甚至限于线性管道, 模型是相当强大的。在 Unix 操作系统是众所周知的[2]的情况下, 通过筛选和管道方法的成功, 已经有力地证明了这种能力, 即管道在霍尔的纸张之前。在一个内部贝尔实验室的备忘录日期 1964年10月11日, Doug 麦克罗伊是玩弄的想法, 将成为 Unix 管道: “我们应该有一些耦合程序的方式, 如花园软管-螺丝在另一个环节, 当它成为必要的按摩数据以另一种方式.这也是 IO 的方式。[3]

霍尔的通信过程比典型的 Unix shell 管道更通用, 因为它们可以以任意模式连接。事实上, 霍尔给出的一个例子是一个3x3 的进程矩阵, 有点像素数筛子, 可以用一个3x3 的正方形矩阵乘以一个向量。

当然, Unix 的管道机制不需要线性布局;只有 shell 语法才行。麦克罗伊报告玩弄与一般管道的外壳早期的语法, 但不喜欢的语法足以实现它 (个人沟通, 2011)。后来的炮弹确实支持一些限制形式的非线性管道。Rochkind 的2dsh 支持达吉斯坦;汤姆. 达夫的 rc 支持树木。

霍尔的语言新颖而有影响力, 但缺乏几个关键方面。主要的缺点是用于通信的缓冲通道不是一流的对象: 它们不能存储在变量中, 不能作为参数传递给函数, 也不能通过通道发送。因此, 通信结构必须在编写程序时进行修复。因此, 我们必须编写一个程序来打印前1000素数, 而不是第一个n素数, 并将一个向量乘以一个3x3 矩阵而不是一个nxn矩阵。

平底锅和 Promela

在 1980年, 仅仅两年后, 霍尔的论文, 杰拉德 ast-holzmann 和 Rob 派克创建了一个协议分析器称为 pan, 以 CSP 方言作为输入。Pan 的 CSP 方言有串联、选择和循环, 但没有变量。即便如此, ast-holzmann 报告说, “潘在1980年11月21日的贝尔实验室数据交换控制协议中发现了第一个错误。[14]. 这种方言很可能是贝尔实验室的第一个 csp 语言, 它无疑提供了使用和实施类似 csp 的语言的经验, 他的第一个许多。

ast-holzmann 的协议分析器开发成了自旋模型检查器及其 Promela 语言, 它以与 Newsqueak (q.v.) 相同的方式具有一流的通道。

Newsqueak

在一个不同的方向移动, 卢卡 Cardelli 和罗布派克开发的想法在 CSP 到吱吱声迷你语言[4] , 以生成用户界面代码。(此吱吱声与 Smalltalk 实现的吱吱声不同。派克后来扩展到了完全成熟的编程语言 Newsqueak [5][6] , 其中生计划9的 Alef [7] [8], 地狱的地狱边缘[9], 和 Google Go [13]。Newsqueak 的主要语义优势是, Newsqueak 将通信信道视为头等对象: 与 CSP 和吱吱声不同, 通道可以存储在变量中, 作为参数传递给函数, 并通过通道发送。这反过来又使通信结构的编程建设, 从而允许创建更复杂的结构比是合理的手工设计。特别是, Doug 麦克罗伊演示了如何使用 Newsqueak 的通信设备来编写用于操作符号电源系列[10]的优雅程序。传统语言的类似尝试往往会陷入簿记的泥潭。在类似的情况下, Rob 派克演示了如何使用通信设备来打破公共事件的编程模型, 编写并发窗口系统[11].

alef

Alef [7] [8]是由菲尔温特伯顿设计的一种语言, 用于将 Newsqueak 思想应用于一个完整的系统编程语言。Alef 有两种类型的, 我们一直在调用进程: 触发和线程。程序被组织成一个或多个触发, 这是可以先发制人调度的共享内存操作系统进程。每个进程都包含一个或多个任务, 它们是协作安排的协同。请注意, 每个任务都分配给一个特定的进程: 它们不会在触发之间迁移。

触发的主要用途是提供可以独立于主要任务来阻止 i/o 的上下文。(计划9没有选择调用, 即使在 Unix 上, 如果要与非网络 i/o 进行重叠计算, 也需要多个触发。Acme 文件[12]对触发和线程有一个很好的简短讨论, 如下面所述,关于计划9窗口系统的演讲说明。

地狱

地狱操作系统是一个计划9剥离的准备机顶盒。它的编程语言, 地狱边缘[9], 受 Alef 的严重影响。它消除了触发和任务之间的区别, 实际上只是触发, 虽然它们的重量比大多数人认为的过程轻。所有并行性都是抢占式的。有趣的是, 尽管这样, 语言没有提供真正的锁定支持。相反, 信道通信通常提供足够的同步, 并鼓励程序员安排任何数据块始终有明确的所有者。显式锁定是不必要的。

Libthread

在计划9世界中, Alef 编译器结果很难维护, 因为计划9被移植到更多的体系结构中。Libthread 最初是为了将 Alef 程序创建到 C, 以便 Alef 编译器可以退役。Alef 的触发和任务被称为触发和线程在 libthread。手动页是最终引用。

Go

Rob 派克和 Ken 汤普森移动到谷歌, 并放置在Go 语言的并发支持中心的 CSP。

入门

要获得模型的感觉, 特别是进程和线程如何交互, 值得阅读 Alef 用户指南[8]。此演示文稿的前三十张幻灯片很好地介绍了 Alef 构造在 C 中的表示方式。

CSP 模型的威力的最佳例子是麦克罗伊和派克的论文, 上面提到[10] [11].

罗布派克的主页包含了关于并发编程的课程的讲义:简介, 以及有关上述两篇文章的幻灯片:斜视和窗口系统。最后的三提供了一个很好的例子, 说明计划9程序通常如何使用触发和任务。

罗布派克给了一个技术讲座在谷歌提供了一个很好的介绍 (57 分钟的视频)。

罗布派克的一半, 他的2010 谷歌 i/o 谈话与拉斯考克斯演示如何使用通道和 Go 的并发来实现负载平衡工作管理系统。
